* [Modern, object-oriented language](https://learn.microsoft.com/en-us/dotnet/csharp/tour-of-csharp/)
* Created by [Anders Hejlsberg](https://en.wikipedia.org/wiki/Anders_Hejlsberg) at Microsoft in the early 2000s
* Evolved from purely OOP to [multi-paradigm](https://learn.microsoft.com/en-us/dotnet/csharp/programming-guide/)
* [Type safe](https://learn.microsoft.com/en-us/dotnet/standard/base-types/common-type-system), [garbage collected](https://learn.microsoft.com/en-us/dotnet/standard/garbage-collection/), and cross-platform

---

### Motivations Behind C#
* Combine C++ power with Visual Basic productivity
* [Truly object-oriented language](https://learn.microsoft.com/en-us/dotnet/csharp/fundamentals/object-oriented/) from the ground up
* Avoid manual memory management
* Provide a language optimized for the CLR
* Provide type safety
* Elegant syntax
* Balance performance needs with developer productivity


---


### Compared with VB

* **Syntax**: More concise (and more cryptic)
  - Curly braces instead of `End` statements
  - Semicolons required
  - Case-sensitive
* **Modern Features**: [Pattern matching](https://learn.microsoft.com/en-us/dotnet/csharp/fundamentals/functional/pattern-matching), [records](https://learn.microsoft.com/en-us/dotnet/csharp/language-reference/builtin-types/record)
* **Community**: Larger ecosystem, more resources
* **Future**: Primary focus of .NET development



---

### Modern C#
* Functional programming features:
  - Higher order functions
  - [Pattern matching](https://learn.microsoft.com/en-us/dotnet/csharp/fundamentals/functional/pattern-matching) and [deconstruction](https://learn.microsoft.com/en-us/dotnet/csharp/fundamentals/functional/deconstruct)
  - [Expression-bodied members](https://learn.microsoft.com/en-us/dotnet/csharp/programming-guide/statements-expressions-operators/expression-bodied-members)
* Modern features focus on developer productivity:
  - [Null reference types](https://learn.microsoft.com/en-us/dotnet/csharp/nullable-references)
  - [Records](https://learn.microsoft.com/en-us/dotnet/csharp/language-reference/builtin-types/record) for immutable data

## .NET From 30,000 Feet

---

### Motivations Behind .NET
* Unify disparate Windows programming models
* Platform-independent [runtime environment](https://learn.microsoft.com/en-us/dotnet/standard/clr)
* Simplify enterprise application development
* Memory safety
* Support multiple programming languages on same platform
* Respond to Java's success

---

### History

* **2002**: [.NET Framework 1.0](https://learn.microsoft.com/en-us/dotnet/framework/get-started/net-framework-versions) released with Visual Studio .NET
* **2002-2015**: [Multiple .NET Framework versions](https://learn.microsoft.com/en-us/dotnet/framework/migration-guide/versions-and-dependencies) (1.0-4.6) with Windows-only focus
* **2016**: [.NET Core 1.0](https://devblogs.microsoft.com/dotnet/announcing-net-core-1-0/) launched as open-source cross-platform redesign
* **2020**: [.NET 5](https://devblogs.microsoft.com/dotnet/announcing-net-5-0/) unification began, dropping "Core" and "Framework" labels
* **2021-Present**: [.NET 6](https://devblogs.microsoft.com/dotnet/announcing-net-6/) and [.NET 7](https://devblogs.microsoft.com/dotnet/announcing-dotnet-7/) established predictable release cadence

---

### .NET Core

* **Open Source**: [Available on GitHub](https://github.com/dotnet/core) under MIT license
* **Source Transparency**: View [runtime source code](https://source.dot.net/) for everything
* **Modular Architecture**: [NuGet-based packages](https://learn.microsoft.com/en-us/dotnet/core/introduction#modular-design) vs monolithic Framework
* **Cross-Platform**: Runs on [Windows, Linux, macOS](https://learn.microsoft.com/en-us/dotnet/core/introduction#cross-platform)
* **Container-Ready**: [First-class Docker support](https://learn.microsoft.com/en-us/dotnet/core/docker/introduction) and small deployment footprint

---

### .NET Framework, .NET Core, .NET 5+ 

* [.NET Framework](https://learn.microsoft.com/en-us/dotnet/framework/get-started/): Legacy, Windows-only. No longer receives new features
* [.NET Core](https://learn.microsoft.com/en-us/dotnet/core/introduction): Cross-platform redesign launched in 2016
* [.NET 5+](https://learn.microsoft.com/en-us/dotnet/core/whats-new/dotnet-5) Core, dropping "Core" from name
* [.NET releases](https://learn.microsoft.com/en-us/dotnet/core/releases/) now follow predictable annual November schedule

### What's the CLR?
* [Common Language Runtime](https://learn.microsoft.com/en-us/dotnet/standard/clr)
* Source code compiles to IL (Intermediate Language), not native binary
* IL is compiled Just-in-time (JIT) to native code
* Key advantages:
  - Platform independence
  - Multi-Language
  - Runtime services - GC, reflection, dynamic loading



---

### Ahead-of-Time (AoT) Compilation
* [Compiles IL to native code](https://learn.microsoft.com/en-us/dotnet/core/deploying/native-aot/) at build time
* Perfect for containerized applications:
  - Faster startup for short-lived instances
  - Smaller deployment size
* Key tradeoff: startup speed vs flexibility
* Limitations: restricted reflection and dynamic code

## IL

---

### Overview
* [IL](https://learn.microsoft.com/en-us/dotnet/standard/managed-code) is .NET's platform-independent bytecode
* C# compiles to IL, not machine code
* Key features:
  - [Stack-based](https://learn.microsoft.com/en-us/dotnet/standard/managed-code#stack-based-execution)
  - [Type-safe](https://learn.microsoft.com/en-us/dotnet/standard/managed-code#type-safety)
  - [Self-describing](https://learn.microsoft.com/en-us/dotnet/standard/managed-code#metadata)
* [JIT](https://learn.microsoft.com/en-us/dotnet/standard/managed-code#just-in-time-compilation) compiles IL to native code at runtime


---

### IL Tools
* Compiled assemblies can be inspected:
  - [**ILDasm**](https://learn.microsoft.com/en-us/dotnet/core/additional-tools/ildasm-tool): Non-graphical, built in
  - [**ILSpy**](https://github.com/icsharpcode/ILSpy) or [dotnet IL tools](https://learn.microsoft.com/en-us/dotnet/core/additional-tools/il-tools)
* Interesting to look at. 
* Good for learning.
* Rarely useful for runtime performance analysis.

## Projects and Solutions

---

### Overview

* **Solution** - Container for multiple projects
* **Project** - Contains source code and configuration
* **Project Types**:
  - [Class Library](https://learn.microsoft.com/en-us/dotnet/core/tutorials/library-with-visual-studio)
  - [Web Application](https://learn.microsoft.com/en-us/aspnet/core/tutorials/first-web-api)
  - [Console Application](https://learn.microsoft.com/en-us/dotnet/core/tutorials/with-visual-studio-code)
* **Dependencies** managed via [NuGet](https://learn.microsoft.com/en-us/nuget/what-is-nuget)

---

### Assembly

* **Theoretical**: [Deployable unit of _compiled_ code](https://learn.microsoft.com/en-us/dotnet/standard/assembly/)
* **Reality**: 
  - Rarely deployed independently
  - Key for code organization
* **No Tree-Shaking**: Unused code included in deployment

---

### NuGet

* **Package Manager** for .NET ecosystem
* Inspired by RubyGems (2010)
* Key features:
  - [Central package repository](https://www.nuget.org/)
  - Version management
  - Dependency resolution
* [Package format](https://learn.microsoft.com/en-us/nuget/reference/nuspec) is open standard
* Integrated into Visual Studio and dotnet CLI

## Tooling

---

### Overview
* **IDEs**: [Visual Studio](https://visualstudio.microsoft.com/), [VS Code](https://code.visualstudio.com/), and [Rider](https://www.jetbrains.com/rider/)
* **CLI Tools**: [.NET CLI](https://learn.microsoft.com/en-us/dotnet/core/tools/) cross-platform development

### Editors

* **Visual Studio**: Full-featured IDE
  - [Rich debugging](https://learn.microsoft.com/en-us/visualstudio/debugger/)
  - Integrated NuGet, testing, deployment

* **JetBrains Rider**: Cross-platform IDE

---

### Dotnet CLI

* **Command-line tool** for .NET development
* Inspired by [Node.js](https://nodejs.org/)/npm
* [Cross-platform](https://learn.microsoft.com/en-us/dotnet/core/tools/)
* Essential for:
  - CI/CD and containers
  - Server environments
  - Automation

---

### Example: Create a new project

```shell
# Create a new empty solution
dotnet new sln --name LS.Enfarculation

# Create a class library project
dotnet new classlib --name LS.Enfarculation.Core

# Add the project to the solution
dotnet sln LS.Enfarculation.sln add LS.Enfarculation.Core/LS.Enfarculation.Core.csproj
```



---

### Dotnet Watch *

* **Hot reload** for test/content
* [Not built into Visual Studio](https://learn.microsoft.com/en-us/dotnet/core/tools/dotnet-watch#dotnet-watch-test)
* Key features: Auto-runs tests / build on file changes
* Limitations:
  - Can be slow with large test suites
  - Sometimes misses changes
  - No test coverage updates
